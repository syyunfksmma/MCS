ChatGPT의 말:

이 에러는 GitHub Actions 워크플로 파일(.github/workflows/ci.yml)이 if 조건에서 secrets를 잘못 참조해서 빌드가 실패한 것을 뜻합니다.
구체적으로 보면 다음과 같습니다.

1. 에러 원인

에러 메시지:

yaml
(Line: 80, Col: 13): Unrecognized named-value: 'secrets'...
(Line: 88, Col: 13): Unrecognized named-value: 'secrets'...

  • if: ${{ secrets.REGISTRY_USERNAME != '' && secrets.REGISTRY_PASSWORD != '' }}

  • if: ${{ secrets.REGISTRY_USERNAME != '' && secrets.REGISTRY_PASSWORD != '' }}

위처럼 if: 조건문 안에서 secrets 객체를 직접 비교(!= '')하고 있는데,
GitHub Actions의 if: 표현식에서 secrets 객체는 문자열 비교 연산이 바로 불가합니다.
secrets는 보안 처리를 위해 단순 값 조회만 허용되고, 조건식 안에서 != '' 같은 비교를 할 수 없습니다.

2. 해결 방법

✅ 권장 수정 예시

yaml
- name: Log in to registry
  if: ${{ secrets.REGISTRY_USERNAME && secrets.REGISTRY_PASSWORD }}
  uses: docker/login-action@v3
  with:
    registry: ${{ env.REGISTRY }}
    username: ${{ secrets.REGISTRY_USERNAME }}
    password: ${{ secrets.REGISTRY_PASSWORD }}

- name: Push Docker image
  if: ${{ secrets.REGISTRY_USERNAME && secrets.REGISTRY_PASSWORD }}
  working-directory: web/mcs-portal
  run: |
    docker tag ${{ env.LOCAL_IMAGE }} ${{ env.REGISTRY_IMAGE }}:${{ github.sha }}
    docker push ${{ env.REGISTRY_IMAGE }}:${{ github.sha }}

🔑 secrets.REGISTRY_USERNAME && secrets.REGISTRY_PASSWORD
→ 둘 다 값이 설정돼 있으면 true, 없으면 false로 간주.
문자열 비교(!= '')가 아니라 존재 여부만 체크해야 정상 동작합니다.

3. 참고 및 베스트 프랙티스

  • 조건문에서 != '' 같은 문자열 비교는 피하기
    → secrets는 값 자체를 문자열 비교할 수 없기 때문입니다.

  • secrets 값이 꼭 필요하다면 env로 옮겨서 사용하거나
    env: 블록에서 한번 변환 후 조건식에서 비교하는 방법도 있습니다.

  • GitHub 공식 문서: Expressions - GitHub Docs

  • 해결: if: ${{ secrets.REGISTRY_USERNAME && secrets.REGISTRY_PASSWORD }} 처럼 존재 여부로 조건 체크.

2. 영향

  • 타입 오류(TypeScript compile error)

  • useRoutingDetail 훅이 routingId를 못 받으므로 API 호출이 안 되고, 모달이 제대로 데이터를 가져오지 못함.

3. 해결 방법

컴포넌트 호출 코드를 수정해야 합니다.
예를 들어:

수정 전

tsx
<RoutingDetailModal
  open={isRoutingDetailOpen && Boolean(selectedRouting)}
  routing={selectedRouting}
  onClose={handleRoutingDetailClose}
  onTabChange={handleRoutingDetailTabChange}
/>

수정 후

tsx
<RoutingDetailModal
  open={isRoutingDetailOpen && Boolean(selectedRouting)}
  routingId={selectedRouting?.id}
  fallbackRouting={selectedRouting}
  onClose={handleRoutingDetailClose}
  onTabChange={handleRoutingDetailTabChange}
/>

이렇게 하면:

  • routingId로 useRoutingDetail 훅이 올바르게 동작

  • 선택된 라우팅 정보가 없을 때는 fallbackRouting이 대체 역할을 하게 됩니다.

4. 정리

  • 무엇: PR 리뷰 봇이 ExplorerShell.tsx 코드의 prop 이름 오류를 지적.

  • 이유: RoutingDetailModal은 routingId와 fallbackRouting을 요구하지만 routing을 전달하고 있었음.

  • 조치: routing={selectedRouting} → routingId={selectedRouting?.id} 및 fallbackRouting={selectedRouting} 로 수정.

이 경고는 HTTPS로 리다이렉트하려는데(UseHttpsRedirection), 앱이 어느 포트로(https 포트) 보내야 할지 몰라서 발생합니다. 즉, HTTPS 엔드포인트(포트)가 설정되지 않았거나 찾을 수 없는 상태입니다.

왜 뜨나요?

  • Program.cs에 app.UseHttpsRedirection();가 있는데,

  • 실행 프로필(launchSettings.json) 또는 Kestrel 설정에 HTTPS 주소/포트가 없거나,

  • 개발용 HTTPS 인증서가 미설치/미신뢰 상태라서 https 포트 추론 실패.

빠른 해결 (권장)

1) launchSettings.json에 https 추가

src/MCMS.Api/Properties/launchSettings.json에 사용 중인 프로필에서 applicationUrl을 http+https 둘 다 명시합니다.

json
{
  "profiles": {
    "MCMS.Api": {
      "commandName": "Project",
      "dotnetRunMessages": true,
      "launchBrowser": true,
      "applicationUrl": "https://localhost:7242;http://localhost:5229",
      "environmentVariables": {
        "ASPNETCORE_ENVIRONMENT": "Development",
        "ASPNETCORE_URLS": "https://localhost:7242;http://localhost:5229",
        "ASPNETCORE_HTTPS_PORT": "7242"
      }
    }
  }
}

핵심: **https 주소와 ASPNETCORE_HTTPS_PORT**를 같이 넣어주면 미들웨어가 포트를 정확히 알 수 있습니다.

2) 개발용 HTTPS 인증서 신뢰(Windows)

PowerShell에서:

powershell
dotnet dev-certs https --check
dotnet dev-certs https --trust

(에러가 나오면 --clean 후 다시 --trust)

3) 재실행

powershell
dotnet run

로그에 Now listening on: https://localhost:7242 같은 줄이 보여야 합니다.

  • Codex GitHub 앱이 리포지토리에 설치돼 있어야 함
